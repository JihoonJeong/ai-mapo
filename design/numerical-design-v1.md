## 0. 설계 철학

### 데이터 → 게임 변환 원칙
1. **실제 데이터가 초기값이다.** 16개 동의 인구, 사업체, 재정은 현실 그대로 시작한다.
2. **시간을 압축한다.** 현실에서 4년 걸릴 변화를 48턴 안에 체감하게 한다. 가속 계수(Acceleration Factor)를 명시.
3. **수식은 숨기되 결과는 보여준다.** Fog of Formulas — AI 자문관에게는 관측 데이터만, 플레이어에게는 대시보드.
4. **재정 압박이 자연 난이도다.** 인위적 불이익 없이 현실 재정 구조(자립도 28%)가 난이도를 만든다.

### 1.2 인구 변동 모델

턴마다 각 동의 인구가 변한다. 세 가지 요인:

```
ΔPop[d][t] = Natural[d][t] + Migration[d][t] + Displacement[d][t]
```

#### (A) 자연 변동 (Natural)
출생률 - 사망률. 마포구 실제 추세 반영.

```
Natural[d][t] = Pop[d] × NaturalRate × (1/4)

NaturalRate = -0.002/년 (서울 평균, 자연감소 추세)
```

**게임 가속**: ×1 (가속 안 함). 자연 변동은 느려서 가속해도 의미 없고, 4년간 누적 효과로 체감.
분기당 약 -178명 (마포구 전체). 동별로는 인구 비례 배분.

#### (B) 전입/전출 (Migration)
플레이어의 정책에 반응하는 핵심 변수.

```
Migration[d][t] = Pop[d] × MigrationPull[d][t] × AccelFactor

MigrationPull[d][t] = Σ(Factor_i × Weight_i) - PushFactors
```

**Pull Factors** (유입 요인):
| 요인 | 가중치 | 설명 |
|------|--------|------|
| 일자리 접근성 | 0.30 | 동 내 종사자수 + 인접 동 종사자수 × 접근 계수 |
| 주거 매력도 | 0.25 | 주거 만족도 - 임대료 부담 |
| 생활 인프라 | 0.20 | 교통 + 상업 + 문화 시설 |
| 안전·환경 | 0.15 | 치안, 공원, 청결도 |
| 교육 여건 | 0.10 | 학교, 학원, 대학 |

**Push Factors** (유출 요인):
| 요인 | 설명 |
|------|------|
| 임대료 상승 | 임대료 > 임계값이면 유출 가속 |
| 소음·혼잡 | 생활인구/상주인구 비율이 높으면 주거 불만 |
| 만족도 하락 | 만족도 < 40이면 유출 가속 |

**AccelFactor = 4.0** (게임 가속)
- 현실: 마포구 연간 순이동 약 ±1~2%
- 게임: 분기당 최대 ±2% 변동 가능 → 4년간 눈에 보이는 인구 이동

**클램프**: 단일 턴 최대 변동 = 동 인구의 ±3%

#### (C) 강제 이주 (Displacement)
재개발 이벤트 등 특수 상황에서만 발생. 이벤트 시스템에서 정의.

### 2.2 사업체 변동 모델

```
ΔBiz[d][t] = NewBiz[d][t] - ClosedBiz[d][t]

NewBiz[d][t] = BaseBiz[d] × DemandFactor[d][t] × PolicyBonus[d][t]
ClosedBiz[d][t] = Biz[d] × (BaseCloseRate + RentPressure[d][t] + CompetitionPressure[d][t])
```

#### 기본 창업/폐업률
```
BaseNewRate = 0.02/분기 (사업체의 2%에 해당하는 신규 창업)
BaseCloseRate = 0.018/분기 (1.8% 자연 폐업)
→ 순증 +0.2%/분기 (연 0.8% 성장, 서울 평균 수준)
```

#### 수요 요인 (DemandFactor)
```
DemandFactor[d] = 0.4 × Norm(생활인구_낮[d]) + 0.3 × Norm(상주인구[d]) + 0.2 × Norm(교통접근성[d]) + 0.1 × 인접동파급
```
생활인구 데이터 확보 완료. 평일 낮 생활인구가 사업체 수요의 핵심 드라이버 (40% 가중치).

#### 임대료 압력 (RentPressure) — 젠트리피케이션 메카닉
```
RentPressure[d][t] = max(0, (상권활력[d][t] - RentThreshold) × RentSensitivity)

RentThreshold = 70 (상권활력이 70을 넘으면 임대료 상승 압력 시작)
RentSensitivity = 0.005 (상권활력 1포인트당 추가 폐업률 0.5%)
```

**핵심 메카닉**: 상권이 뜨면 → 임대료 상승 → 영세 사업체 폐업 → 프랜차이즈 진입 → 상권 특색 감소.

```
상권특색[d][t+1] = 상권특색[d][t] × (1 - FranchiseRate[d][t] × 0.02)
FranchiseRate[d][t] = 누적 RentPressure에 비례
```

## 3. 재정 모델

### 3.1 초기 상태 (실제 데이터)

```
연간 총예산: ~8,750억원
분기 예산: ~2,188억원/턴
재정자립도: 28%
```

### 3.2 세입 구조

```
분기세입[t] = 지방세[t] + 조정교부금[t] + 보조금[t] + 세외수입[t]
```

| 세입 항목 | 연간 추정 | 분기 추정 | 플레이어 영향 가능? |
|----------|----------|----------|-----------------|
| 지방세 (재산세 등) | ~2,450억 | ~613억 | ◎ 상권 활성화 → 사업체 증가 → 세수 증가 |
| 조정교부금 (서울시) | ~2,800억 | ~700억 | △ 간접적 (인구 유지 → 교부금 배분) |
| 국고·시비 보조금 | ~3,000억 | ~750억 | △ 특정 사업 유치 시 추가 확보 |
| 세외수입 | ~500억 | ~125억 | ○ 시설 운영, 과태료 등 |
| **합계** | **~8,750억** | **~2,188억** | |

**지방세 산출 로직**:
```
지방세[t] = BaseTax × (1 + TaxGrowthRate[t])

TaxGrowthRate[t] = f(
  Δ사업체수,     // 사업체 증가 → 세수 증가
  Δ종사자수,     // 고용 증가 → 지방소득세 간접 기여
  부동산가격변동,  // 공시지가 → 재산세  [DATA_NEEDED: 부동산]
  경기변동        // 외부 이벤트
)
```

**핵심 제약**: 마포구가 직접 올릴 수 있는 세금이 거의 없다. 재산세율은 법정, 취득세는 서울시. 세수를 늘리려면 간접적으로 — 사업체를 늘리고, 부동산 가치를 올리고, 인구를 유지해야 한다. 이것이 게임의 근본 딜레마.

### 3.3 세출 구조 (예산 배분)

매 턴 플레이어가 분기 예산을 배분한다.

**의무 지출 (잠금)**:
```
복지 의무지출 = 분기예산 × 0.55 (약 1,203억)
인건비·기본운영 = 분기예산 × 0.15 (약 328억)
→ 자유 배분 가능 예산 = 분기예산 × 0.30 (약 656억)
```

현실 데이터가 복지 55%이면 그대로 간다 (JJ 결정). 복지 55% 안에서도 배분 선택지 존재 — 어르신 돌봄 vs 청년 지원 vs 보육 강화 등. 자유 예산 30%가 구청장의 진짜 재량이며, 이 빡빡함이 자연 난이도다.

**자유 배분 카테고리** (약 656억/턴):
| 카테고리 | 투자 효과 | 시차 |
|---------|---------|------|
| 경제·일자리 | 사업체 유치, 창업 지원, 상권 활성화 | 2~4턴 |
| 교통·인프라 | 도로 정비, 자전거도로, 주차장 | 3~6턴 |
| 문화·관광 | 축제, 문화시설, 관광 인프라 | 1~3턴 |
| 환경·안전 | 공원, CCTV, 쓰레기 관리, 소음 대책 | 1~2턴 |
| 교육·보육 | 어린이집, 방과후학교, 도서관 | 4~8턴 |
| 주거·복지 (추가) | 임대주택, 어르신 돌봄, 청년 지원 | 2~6턴 |
| 도시재생 | 재개발, 리모델링, 마을 만들기 | 6~12턴 |

**예산 효율 체감 감소**: 같은 카테고리에 집중 투자하면 효율 감소.
```
EffectiveSpend = RawSpend × (1 / (1 + 0.3 × (RawSpend / OptimalSpend - 1)))
OptimalSpend = 카테고리별 기본 적정 예산 (전체의 약 5~15%)
```

### 3.4 관광-세수 되먹임 루프 (핵심 순환)

```
관광 활성화
  → 유동인구 증가 [DATA_NEEDED: 생활인구]
  → 숙박음식업 매출 증가
  → 사업체 수 증가 (숙박음식, 도매소매)
  → 지방세 증가 (사업체 기반)
  
  BUT 동시에:
  → 임대료 상승 (RentPressure ↑)
  → 소상공인 폐업 (영세 사업체 이탈)
  → 상권 특색 감소 (프랜차이즈화)
  → 관광 매력 감소
  → 유동인구 감소
  → 루프 역전
```

**수식화**:
```
관광매력[d][t] = 문화시설점수[d] × 상권특색[d][t] × 접근성[d] × 안전환경[d]
유동인구[d][t] = f(관광매력[d][t], 교통접근성[d], 계절, 이벤트)
상권매출[d][t] = 유동인구[d][t] × 객단가 × 소비성향
임대료압력[d][t] = f(상권매출[d][t], 사업체밀도[d][t])
사업체생존율[d][t] = BaseRate - f(임대료압력[d][t], 사업체규모)
상권특색[d][t+1] = f(독립사업체비율[d][t], 문화다양성[d][t])
```

이 루프의 균형점을 찾는 것이 플레이어의 핵심 과제. 개입 수단:
- 임대료 안정화 정책 (지원금, 상생 협약)
- 특색 상권 보호 (업종 다양성 인센티브)
- 관광 분산 (서교동 → 망원동, 성산동으로 유도)

## 4. 만족도 모델

### 4.1 구조

만족도는 **동별 × 생애주기별**로 계산. 0~100 스케일.

```
Satisfaction[d][age][t] = Σ(Component_i[d][t] × Weight_i[age])
```

### 4.2 만족도 구성 요소 (6개)

| 구성 요소 | 산출 방식 | 범위 |
|----------|---------|------|
| 경제·일자리 | f(사업체수, 실업률, 소득수준) | 0~100 |
| 교통·접근성 | f(지하철승하차, 버스노선, 혼잡도) | 0~100 |
| 주거·환경 | f(임대료부담, 공원면적, 소음, 청결) | 0~100 |
| 안전 | f(범죄율, CCTV, 가로등) | 0~100 |
| 문화·여가 | f(문화시설, 상권특색, 커뮤니티) | 0~100 |
| 복지·돌봄 | f(복지시설, 의료접근, 보육) | 0~100 |

### 4.3 생애주기별 가중치

| 구성요소 | 청년(20-34) | 중장년(35-49) | 장년(50-64) | 고령(65+) |
|---------|-----------|-------------|-----------|---------|
| 경제·일자리 | **0.30** | 0.25 | 0.20 | 0.10 |
| 교통·접근성 | 0.20 | 0.15 | 0.15 | 0.15 |
| 주거·환경 | 0.15 | **0.25** | 0.20 | 0.15 |
| 안전 | 0.10 | 0.15 | 0.15 | 0.15 |
| 문화·여가 | **0.20** | 0.10 | 0.15 | 0.15 |
| 복지·돌봄 | 0.05 | 0.10 | 0.15 | **0.30** |

영유아·청소년은 부모(중장년) 만족도를 공유.

### 4.4 초기 만족도 설정

게임 시작 시 모든 동의 만족도를 **60** (보통)으로 세팅하되, 데이터 기반 보정:

```
초기만족도[d] = 60 + 경제보정[d] + 교통보정[d] + 주거보정[d]
```

| 보정 요인 | 계산 | 범위 |
|----------|------|------|
| 경제보정 | (종사자비율[d] - 구평균) × 10, 클램프 ±10 | -10 ~ +10 |
| 교통보정 | (일승하차합[d역] / 50000) × 5, 클램프 ±8 | -8 ~ +8 |
| 주거보정 | (고령비율[d] < 18% ? +3 : -2) | -2 ~ +3 |

예시:
- 서교동: 60 + 10(경제↑) + 8(홍대역) - 2(혼잡) = **76** → 경제는 좋지만 주거 스트레스
- 염리동: 60 - 5(경제↓) - 3(교통↓) + 3(주거↑) = **55** → 조용하지만 불편
- 상암동: 60 + 8(경제↑) + 5(DMC역) + 2(공원) = **75** → 양호

(정확한 초기값은 데이터 보정 후 확정)

### 4.5 만족도 변동

```
ΔSatisfaction[d][age][t] = PolicyEffect[d][t] + EventEffect[d][t] + SpilloverEffect[d][t] + Decay[d][t]

Decay = -0.5/턴 (아무것도 안 하면 자연 하락, 인프라 노후화)
```

만족도는 인구 이동의 핵심 드라이버:
- 만족도 > 70: 유입 경향
- 만족도 40~70: 안정
- 만족도 < 40: 유출 가속, 사업체 이탈 시작

## 5. 동간 파급효과 (Spillover)

### 5.1 인접 행렬

마포구 16개 동의 물리적 인접 관계. 인접한 동끼리 정책/변화가 전파.

```
인접계수 = 1.0 (직접 인접), 0.3 (2차 인접), 0 (비인접)
```

**핵심 인접 클러스터**:

| 클러스터 | 구성 동 | 특성 |
|---------|--------|------|
| 홍대 벨트 | 서교동 ↔ 합정동 ↔ 연남동 ↔ 망원1동 | 관광·상권 연동. 서교동 임대료 ↑ → 합정·연남으로 전파 |
| 공덕 업무권 | 공덕동 ↔ 도화동 ↔ 아현동 ↔ 용강동 | 업무지구 + 주거. 재개발 효과 공유 |
| 성산-상암 | 성산1동 ↔ 성산2동 ↔ 상암동 | DMC + 주거 + 공원. 독립적 생활권 |
| 서강-대흥-신수 | 서강동 ↔ 대흥동 ↔ 신수동 ↔ 염리동 | 대학가 + 주거. 중간 지대 |

### 5.2 파급 메카닉

```
Spillover[d][t] = Σ(AdjCoeff[d][d'] × ΔValue[d'][t] × SpilloverRate)
```

| 파급 유형 | SpilloverRate | 설명 |
|----------|-------------|------|
| 임대료 전파 | 0.3 | 서교동 임대료 10% 상승 → 합정동 3% 상승 |
| 관광 분산 | 0.2 | 서교동 관광매력 포화 → 연남동에 20% 유입 |
| 범죄 확산 | 0.15 | 소음·치안 문제 인접 동 전파 |
| 인프라 공유 | 0.4 | 공덕역 개선 → 인접 동 교통 만족도 상승 |

**젠트리피케이션 도미노**: 가장 중요한 파급.
```
서교동 상권 과열 → 임대료 ↑ → 소상공인 연남동 이동
→ 연남동 상권 활성화 → 연남동 임대료 ↑ → 망원동 이동
→ 이 파동이 2~4턴 시차로 전파
```

#### 공약 리스크-리워드
- 최소 1개, **최대 4개** 선택
- 달성: **+15점/개**, 미달성: **-20점/개** (실패 페널티 > 성공 보상)
- 4개 전부 달성: +60 (영웅) → S등급 도전 가능
- 4개 중 3개: +45-20 = +25
- 4개 중 2개: +30-40 = **-10** (욕심의 대가)
- 1개만 골라 달성: +15 (안전하지만 평범)
- 0개 선택: 공약 점수 없음, 나머지 70점 만점으로만 평가

자연스러운 전략 분기: 신중한 플레이어는 1~2개, 도전적 플레이어는 3~4개.

## 9. 공약 시스템

### 9.1 구조

게임 시작 시 **8개 후보 중 최소 1개, 최대 4개 공약 선택**.
- 많이 선택할수록 높은 점수 가능하지만, 미달성 시 페널티 (달성 +15, 미달성 -20)
- 각 공약의 달성 조건은 **절대 수치**로 명확히 제시
- 진행 중 대시보드에서 달성률 실시간 추적

### 9.2 공약 후보 (데이터 기반)

| # | 공약 | 달성 조건 | 난이도 | 기반 데이터 |
|---|------|---------|--------|-----------|
| 1 | 홍대 상생 상권 | 서교동 상권특색 > 70 유지, 소상공인 폐업률 < 구평균 | ★★★ | 서교동 사업체 13,643, 임대료 압박 |

| 3 | 청년 정착 도시 | 청년(20-34) 인구 감소율 < 2% (4년간) | ★★★ | 청년 25.3%, 1인가구 밀집 |

- "균형 발전 + DMC 2.0" = 상암 집중 투자 vs 전체 균등 배분

## 부록 B: 수식 요약 (Buddy 구현 레퍼런스)

```javascript
// === 인구 변동 ===
function updatePopulation(district, turn) {
  const natural = district.pop * NATURAL_RATE * (1/4);
  const pull = calcMigrationPull(district, turn);
  const migration = district.pop * pull * ACCEL_MIGRATION * ageMobility[age];
  const clamped = clamp(migration, -district.pop * 0.03, district.pop * 0.03);
  return district.pop + natural + clamped;
}

// === 사업체 변동 ===
function updateBusinesses(district, turn) {
  const newBiz = district.biz * BASE_NEW_RATE * demandFactor(district) * policyBonus(district);
  const closed = district.biz * (BASE_CLOSE_RATE + rentPressure(district) + competition(district));
  return district.biz + newBiz - closed;
}

// === 만족도 ===
function calcSatisfaction(district, ageGroup, turn) {
  const components = [economy, transit, housing, safety, culture, welfare];
  const weights = AGE_WEIGHTS[ageGroup];
  return sum(components.map((c, i) => c(district) * weights[i]));
}

// === 세수 ===
function calcRevenue(turn) {
  const localTax = BASE_TAX * (1 + taxGrowth(turn));
  const grants = BASE_GRANTS * (totalPop(turn) / BASE_POP);
  const subsidies = BASE_SUBSIDIES + projectSubsidies(turn);
  return localTax + grants + subsidies;
}

// === 파급효과 ===
function calcSpillover(district, type, turn) {
  return adjacentDistricts(district).reduce((sum, adj) => {
    return sum + ADJ_COEFF[district][adj] * deltaValue(adj, type, turn) * SPILL_RATE[type];
  }, 0);
}
```

상수값은 플레이테스트에서 조정. 여기 적힌 숫자는 초기 추정치.
